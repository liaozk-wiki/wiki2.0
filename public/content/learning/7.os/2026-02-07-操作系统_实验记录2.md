# 计算机系统安全

安全：

Confidentiality：保密性

Integrity：完整性（不能让别人改）

Availability：可用性（别人不能让我用不了）

- Fork bomb: `:(){ :|: & };:`
- [Algorithmic complexity attack](https://jyywiki.cn/OS/2025/slides/26.1.html); 例子：[Hash table](https://nvd.nist.gov/vuln/detail/CVE-2011-4858)
- Distributed DoS (DDoS)



现代安全的领域：[IEEE S&P (Oakland) 2025](https://www.ieee-security.org/TC/SP2025/cfpapers.html)

软硬件系统协同





OS：访问控制



进程 + 虚拟内存已经实现了隔离

- 进程只能以 ELF 规定的权限访问自己的虚拟地址空间
- **系统调用**是唯一访问操作系统对象的途径
  - (假设，内核没有被漏洞攻破)
  - 操作系统还有虚拟化机制：虚拟机、容器 (cgroups, namespaces, ...)

访问控制：**限制程序对操作系统对象的访问**

- 拒绝越权访问 →→ Confidentiality
- 拒绝越权修改 →→ Integrity
- (再加上公平资源调度 →→ Availability)



操作系统如何实现访问控制？

对象：进程树，目录树，管道，IO设备等各种对象

最简单的方式：我们只需要实现一张表

![image-20251026145100093](https://markdownupload.oss-cn-chengdu.aliyuncs.com/md/image-20251026145100093.png)

越权访问：EACCESS

这张表会变得非常的庞大



进一步优化：通过id简化这张表（uid，gid，mode（rwx））

uid, gid, mode

- uid = 0 →→ root, 其他都是 “普通用户”，访问资源需要经过这张表的控制
  - root 可以访问所有对象
  - 也可以 setuid
  - 子进程继承父进程的 uid
- gid “完全自由” (虽然一般 0 是 root)
  - 一个用户可以属于多个组
- mode: r, w, x 的权限
  - 例子：owner 只写不可读，audit 组可以读的日志文件





**操作系统完全看不到用户名**通过 setuid, setgid, chmod 系统调用实现访问控制 ！

有一个login程序，解析用户名与密码，操作系统是完全不管这个的。etc/passwd ，shadow记录密码hash

操作系统只有uid的概念，不管如何解读用户



uid, gid, mode 并不是实现访问控制的唯一方法：what this is？

- Access Control List (ACL)
  - 基于 xattr 实现，支持为任意数量的用户和组设置权限
- SELinux/AppArmor：传统上文件所有者可以自由决定谁可以访问，现代全局安全策略
  - sudo apparmor_status | ag --gpt -q Explain
- Capabilities：特权分解
  - capsh --drop=cap_net_raw -- -c 'ping 127.0.0.1' (注意这是 fail on execve; getcap 查看 capabilities)



**setuid file**: 通过将二进制文件设置为 setuid 和 root 用户/组，可以实现普通用户以 root 用户身份执行该文件。这个危险的特性带来很多潜在的安全漏洞——因此 bash 在检测到这种行为时，会自动丢弃权限。当一个可执行文件设置了 `setuid` 位后，任何用户运行它时，该进程的“有效用户 ID（effective UID）”将变为该文件所有者的 UID。普通用户可以临时获得文件所有者的权限来执行该程序。

## AM运行

1._start

2._start_c

3.启动cpu初始化后调用main方法，然后在mep_init中启动ap cpu然后user_entry

4.应用cpu每一个都会跳转到0x7c00然后走到_start_c，然后调用user_entry

5.最终所有的cpu都会到达os->run方法

6.然后程序不停的trap，调用__am_irq_handle，调用user_handler，此时所有的cpu都化身为中断处理器：系统不停的中断，cpu不停的响应中断，更改寄存器的值，状态机不停的重置状态然后运行。

task中stack的生命周期整理：

1.初始化时使用task中的stack高位作为context

2.第一次中断时的rsp指向的地址暂时没有确认，但一旦运行到trap程序后，kmt_context_save会将其指引为task_all对应的context，并传递给trap.s中的函数

3.然后每次cpu都是使用的task中的stack了

4.因为每次trap发生的时机不同，实际上task的context地址也会有所变化。



现在的问题变成了：为什么会stack溢出？考虑区分中断stack与任务stack？更进一步，可能需要考虑：观察每次trap时的stack状态，但目前不想去处理了。至此可以认为简单的实现了KMT部分
